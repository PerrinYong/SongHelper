# SongHelper 工程能力手册

本文档是当前工程唯一推荐的人类阅读手册。

目标：

- 说明本工程的定位
- 完整说明当前保留的功能
- 用标题分级同时组织“底层工具”和“上层技能”
- 明确每项能力的输入、输出、用途和边界

---

## 一、工程定位

`SongHelper` 是一个面向歌曲创作、分析、处理与迭代的**音乐工具箱工程**。

它不是 DAW，也不替你接管创作流程。它更像一个“音乐中控层”：

- 你把音频、歌词、参考曲、创作意图交给它
- 它调用合适的底层工具产出分析和处理结果
- 再由我基于这些结果继续做解释、判断、建议和迭代

它的核心价值不是一次性做完一首歌，而是把高频、可复用的工作沉淀成：

- 脚本
- CLI 命令
- 中间结果格式
- 文档与技能模板

---

## 二、工程边界

### 2.1 当前适合做的事

- 音频基础探测
- BPM / 调性 / 段落的粗分析
- 人声或单旋律的主旋律提取
- MIDI / 简谱草稿输出
- 粗分离（HPSS / 中置）
- 音频拼接、简单混响、响度标准化
- 创作策划、风格拆解、结构建议

### 2.2 当前不保留或不夸大的能力

- 不把粗分离说成高质量 AI stem 分离
- 不把 `pyin` 提取结果说成出版级最终乐谱
- 不保留“多次自动处理后再做自动旋律重合判断”这类高误差链能力
- 不声称替代专业混音 / 母带 / 记谱流程

---

## 三、工作目录约定

每首歌放在：`workspace/<作品名>/`

建议子目录：

- `source/`：原始素材
- `analysis/`：分析结果
- `stems/`：分离结果
- `scores/`：MIDI、简谱、旋律 JSON
- `mixes/`：处理中间版本
- `exports/`：导出结果
- `notes/`：人工记录

---

# 第一部分：底层工具层

底层工具层关注“具体执行”。

每个工具都尽量有明确输入、输出和适用边界。

---

## 四、音频探测工具

### 4.1 功能

命令：`probe-audio`

对一个音频文件做基础探测，输出：

- 文件格式
- 采样率
- 声道数
- 时长
- BPM 粗估
- 调性中心粗估
- RMS 均值 / 波动
- 频谱重心均值
- 过零率均值

### 4.2 输入

- 单个音频文件路径

### 4.3 输出

- JSON 文件

### 4.4 有什么用

- 快速总览素材
- 在真正分析前先看音频条件
- 判断后续是否更适合做旋律提取、粗分离或直接音频整理

### 4.5 边界

- BPM 和调性都是粗估，需要耳朵最终确认

---

## 五、歌曲分析工具

### 5.1 功能

命令：`analyze`

对歌曲做较完整的音乐分析，输出：

- BPM
- 调性
- 拍点数量
- 粗段落结构

### 5.2 输入

- 单个音频文件路径

### 5.3 输出

- JSON 分析结果

### 5.4 有什么用

- 看一首歌的大体结构
- 作为后续转写、风格拆解、段落命名的基础

### 5.5 边界

- 段落结构是机器切分结果，不等于人工最终标注

---

## 六、粗分离工具

### 6.1 功能

命令：`rough-separate`

支持两种模式：

#### 6.1.1 `hpss`

- 谐波 / 打击分离
- 输出：`harmonic.wav`、`percussive.wav`

#### 6.1.2 `center`

- 中置声像提取
- 输出：`center_mono.wav`、`side_mono.wav`、`minus_center_stereo.wav`

### 6.2 输入

- 单个音频文件
- 分离模式 `hpss` 或 `center`

### 6.3 输出

- 若干 wav 文件

### 6.4 有什么用

- 先做低成本预处理
- 为后续旋律提取准备一个更干净的输入
- 在没有更强模型时，尝试获得粗略可用的人声中心

### 6.5 边界

- 不是高质量 AI stem 分离
- 只能作为草稿、中间步骤或实验用途

---

## 七、主旋律提取工具

### 7.1 功能

命令：`transcribe`

从单轨旋律音频中提取主旋律，输出：

- MIDI
- 音符 JSON
- 简谱草稿

### 7.2 输入

- 单轨人声或较清晰的旋律音频
- 可选：tempo / tonic / mode 提示

### 7.3 输出

- `melody.mid`
- `melody_notes.json`
- `melody_jianpu.txt`

### 7.4 有什么用

- 从 vocal 得到主旋律草稿
- 为后续清理、人工修谱、MIDI 编辑提供底稿

### 7.5 边界

- 基于 `pyin` 的单旋律估计
- 适合作为草稿，不适合作为最终正式谱

---

## 八、旋律清理工具

### 8.1 功能

命令：`clean-melody`

对提取出来的旋律音符做清理：

- 去除碎音
- 去除异常高音
- 去除孤立跳点
- 合并相邻同音

### 8.2 输入

- `transcribe` 输出的音符 JSON

### 8.3 输出

- 清理后 MIDI
- 清理后简谱
- 清理后音符 JSON

### 8.4 有什么用

- 把机器噪点很多的旋律变成更接近主旋律的版本

---

## 九、简谱与可读谱工具

### 9.1 `midi-to-jianpu`

#### 功能

把 MIDI 转成简谱文本。

#### 输入

- MIDI 文件
- 指定主音与 mode

#### 输出

- 简谱文本

#### 有什么用

- 核对 MIDI
- 把旋律转成更方便阅读的数字谱文本

### 9.2 `human-jianpu`

#### 功能

把清理后的旋律重新排成更适合人阅读的简谱文本。

#### 输入

- 清理后的旋律 JSON

#### 输出

- 人类可读的简谱文本

#### 有什么用

- 比原始事件表更适合人工复核
- 适合继续手工整理成正式简谱

---

## 十、音频整理工具

### 10.1 `concat-audio`

#### 功能

按顺序拼接多个音频文件。

#### 输入

- 多个音频输入文件

#### 输出

- 一个拼接后的音频文件

#### 有什么用

- 拼 demo
- 拼多个段落素材
- 快速做版本草稿

### 10.2 `apply-reverb`

#### 功能

基于 ffmpeg 的 `aecho` 预设添加简单混响。

#### 当前预设

- `small_room`
- `hall`
- `plate`

#### 输入

- 一个音频文件
- 一个预设

#### 输出

- 一个加过混响的音频文件

#### 有什么用

- 快速做空间感尝试
- demo 级润色

### 10.3 `normalize-audio`

#### 功能

基于 ffmpeg `loudnorm` 做响度标准化。

#### 输入

- 输入音频
- 可选目标 `I / TP / LRA`

#### 输出

- 标准化后的音频

#### 有什么用

- 统一不同版本音量
- 做试听版输出

---

## 十一、底层工具的典型串联

### 场景 1：分析一首歌

`probe-audio -> analyze`

### 场景 2：从 vocal 提主旋律

`rough-separate(可选) -> transcribe -> clean-melody -> human-jianpu`

### 场景 3：做 demo 级音频整理

`concat-audio -> apply-reverb -> normalize-audio`

# 第二部分：上层技能层

上层技能层关注“在真实创作场景里要解决什么问题”。

一个上层技能通常会组合多个底层工具，再加上人工判断与音乐解释。

---

## 十二、歌曲分析技能

### 12.1 解决什么问题

当你拿到一首现有歌曲、demo、录音或 AI 生成结果，想先看清它“整体长什么样”，这个技能最合适。

### 12.2 典型需求

- 这首歌大概多少 BPM？
- 是什么调？偏大调还是小调？
- 结构上大概怎么分段？
- 哪些地方更像主歌，哪些地方更像副歌？

### 12.3 常用底层工具

- `probe-audio`
- `analyze`

### 12.4 输出

- 音频探测 JSON
- 分析 JSON
- 文字解释结论

---

## 十三、主旋律转写技能

### 13.1 解决什么问题

当你已经有 vocal、哼唱、单旋律录音，想把它变成更可编辑、更可分析的旋律数据时，用这个技能。

### 13.2 典型需求

- 从人声提主旋律
- 导出 MIDI
- 转成简谱草稿
- 清理掉碎音和异常高音

### 13.3 常用底层工具

- `transcribe`
- `clean-melody`
- `midi-to-jianpu`
- `human-jianpu`

### 13.4 输出

- MIDI
- 音符 JSON
- 简谱草稿
- 清理后主旋律版本

### 13.5 注意事项

- 当前结果更适合作为草稿和分析底稿，不是正式出版谱

---

## 十四、人声与 stem 处理技能

### 14.1 解决什么问题

当你需要把混合音频拆成更便于处理的部分，比如人声、伴奏、中心声像或谐波/打击内容时，用这个技能。

### 14.2 典型需求

- 给一首歌做人声分离
- 提取 center vocal 试着转写
- 先把鼓/打击和谐波拆开做粗分析

### 14.3 常用底层工具

- Demucs 分离（外部模型能力）
- `rough-separate --mode center`
- `rough-separate --mode hpss`

### 14.4 输出

- vocals / no_vocals（若走 Demucs）
- center / side / minus_center
- harmonic / percussive

### 14.5 注意事项

- HPSS 和 center 只是粗处理，不等于高质量 stem 分离

---

## 十五、音频整理与 Demo 处理技能

### 15.1 解决什么问题

当你只是想快速把素材整理成一个可听、可发、可比对的 demo 版本时，用这个技能。

### 15.2 典型需求

- 拼接几个片段
- 给人声或整段音频加一点空间感
- 统一多个版本的响度

### 15.3 常用底层工具

- `concat-audio`
- `apply-reverb`
- `normalize-audio`

### 15.4 输出

- 拼接版音频
- 混响版音频
- 标准化版音频

### 15.5 注意事项

- 适合 demo 级整理，不等于专业混音 / 母带

---

## 十六、创作策划与风格拆解技能

### 16.1 解决什么问题

当你的问题还不在“处理音频”，而是在“这首歌该怎么成立”，这个技能更有用。

### 16.2 典型需求

- 主题怎么立？
- 主歌 / 副歌如何分工？
- 为什么这首歌没有记忆点？
- 应该借鉴参考曲的什么，不该借什么？
- MV、视觉、提示词怎么和音乐统一？

### 16.3 常用底层支撑

- 歌曲分析结论
- 旋律转写结果
- 人工风格判断与文本推理

### 16.4 输出

- 创作方向建议
- 结构建议
- 歌词 / hook 建议
- 风格拆解结论
- 提示词与视觉方向建议

### 16.5 注意事项

- 这是高层创作协作能力，不一定对应单个 CLI 命令，但会以底层分析结果作为证据支撑

---

## 十七、推荐阅读与使用顺序

### 如果你第一次接触工程

1. 先看本手册
2. 再看 `README.md`
3. 再看 `docs/TASK_EXAMPLES.md`

### 如果你准备真正处理一首歌

1. 建立 `workspace/<作品名>/`
2. 放入 `source/`
3. 先 `probe-audio`
4. 再判断是否 `analyze` / `rough-separate` / `transcribe`
5. 最后再进入创作或制作层面的迭代

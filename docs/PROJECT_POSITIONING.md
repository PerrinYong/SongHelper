# 工程定位与演进说明

## 1. 当前工程是什么

`SongHelper` 不是单一脚本，也不是一次性分析目录。

它的定位是：

- 你的歌曲创作辅助工作空间
- 我执行音乐分析、编辑、转写、整理任务时的本地工具箱
- 一个可持续迭代的“音乐创作工程化底座”

同时需要明确：

- 它是工具箱，不是流程接管器
- 你的创作流程、优先级、审美判断与任务节奏由你主导
- 我负责在你需要时提供分析、处理、脚本化与工具化支持

这个仓库既服务当前歌曲，也服务未来的多首作品、多个版本与多类处理任务。

## 2. 当前工程解决什么问题

歌曲创作通常会反复经过以下环节：

- 找方向：风格、情绪、参考曲、叙事视角
- 做 demo：手机录音、钢琴弹唱、临时哼唱
- 拆与听：分析节奏、调性、旋律、和弦、结构
- 提与转：做人声分离、旋律提取、乐谱化、MIDI 化
- 编与改：配器、和声、段落推进、hook 强化
- 混与审：效果、空间、响度、清晰度、参考对比
- 导与存：导出版本、留档、复盘、继续修改

这些环节容易产生以下痛点：

- 文件和版本混乱
- 同类工作重复劳动
- 某次分析或处理难以复现
- 工具散落在不同地方，无法积累
- 每次都要从头定义目录、脚本和流程

当前工程的核心价值，就是把这些工作持续沉淀成：

- 统一目录
- 统一脚本入口
- 统一输出位置
- 可复用工具
- 可追踪文档

## 3. 这个工程在后续合作中的角色

后续你交给我的任务，除了直接完成任务本身，我还可以在执行过程中：

- 创造新工具
- 保存工具
- 为工具补充说明
- 在后续任务中继续复用同一个工具
- 发现通用模式后，把一次性脚本升级为稳定模块

也就是说，这个仓库既是“工作现场”，也是“工具仓库”，但不是替代你主导创作流程的系统。

## 4. 我会如何把它当作工作空间和工具箱

### 4.1 工作空间

用于承接你当前歌曲项目中的过程文件与结果文件：

- `works/`：原始歌曲素材
- `workspace/analysis/`：分析结果、报告、特征提取输出
- `workspace/stems/`：分离后的人声、伴奏、鼓、贝斯等
- `workspace/scores/`：MIDI、简谱、乐谱草稿
- `workspace/mixes/`：处理中间版本、rough mix
- `workspace/exports/`：最终导出物
- `workspace/notes/`：创作说明、任务记录、人工判断

### 4.2 工具箱

用于沉淀通用能力：

- 音频分析脚本
- stem 分离脚本
- 旋律提取与 MIDI 导出脚本
- 音频裁剪、拼接、批量转码工具
- 报告生成器
- 项目初始化与归档工具

初期这些工具可能是轻量脚本；后续可以逐步升级为：

- 标准 CLI 子命令
- 独立模块
- 可配置流水线
- 面向单首作品或整批作品的批处理能力

## 5. 我后续执行任务时的默认策略

当你给我一个新任务时，我默认不仅考虑“做完这次”，也会考虑：

1. 这项工作是否值得沉淀为脚本
2. 这个脚本是否会在后续复用
3. 输出目录是否应该标准化
4. 是否需要生成说明文档方便下次直接继续
5. 是否需要把一次性处理步骤升级为可重复流程

因此，后续合作中我会尽量把：

- 临时分析 → 标准分析脚本
- 单次处理 → 可复用命令
- 口头建议 → 文档化工作流
- 零散文件 → 结构化产物

## 6. 可持续扩展的实现方向

### 6.1 分析能力

- BPM / beat tracking
- key detection
- chord candidate estimation
- melody contour extraction
- section segmentation
- spectral / loudness analysis
- reference track comparison

### 6.2 编辑能力

- trim / split / concat
- fade in / fade out
- loudness normalization
- sample rate / format conversion
- batch export

### 6.3 创作辅助能力

- 歌词结构建议
- hook 优化建议
- 旋律可唱性分析
- 和声编排建议
- 配器层次建议

### 6.4 转写与谱面能力

- vocal pitch extraction
- audio to MIDI
- MIDI to note list
- 简谱草稿输出
- lead sheet 草稿输出

### 6.5 混音辅助能力

- 参考曲对比报告
- 频段拥挤提示
- 空间感与混响建议
- 主唱存在感建议
- revision checklist 自动生成

### 6.6 工程化能力

- 单首歌项目模板
- 批处理任务模板
- 结果归档与命名规范
- 工具注册表
- 任务日志与版本记录

## 7. 后续建议的工具沉淀原则

后续新增工具时，优先遵循以下原则：

- **最小可用**：先解决当前任务，不做空泛大系统
- **可复用**：同类任务下次可直接调用
- **可验证**：工具输出要能检查
- **可组合**：多个小工具可串成流程
- **可文档化**：知道怎么用、产物在哪
- **不破坏原素材**：优先输出到新目录，不覆盖原始文件

## 8. 推荐的未来目录演进

当前已初始化基础骨架；后续可按需要扩展：

- `src/songhelper/analysis/`
- `src/songhelper/audio/`
- `src/songhelper/transcription/`
- `src/songhelper/mixing/`
- `src/songhelper/reports/`
- `src/songhelper/tools/`
- `configs/`
- `scripts/`
- `toolbox/` 或 `presets/`

其中可考虑：

- `tools/`：通用 Python 工具实现
- `scripts/`：面向具体任务的入口脚本
- `configs/`：分析参数、导出参数、效果预设
- `presets/`：混响、导出、项目模板等预设

## 9. 你可以如何使用这个工程

你后续只需要直接下达任务，例如：

- “分析 `works/混天绫/混天绫.wav` 的 BPM、调性和段落结构”
- “给这首歌做人声分离，输出到 `workspace/stems/混天绫/`”
- “从分离后的人声提取主旋律，生成 MIDI”
- “把副歌这一段做两个不同的混响方案”
- “把这次处理写成可复用脚本，后续别的歌也能直接跑”

我会在完成任务的同时，尽量把可沉淀部分留在仓库里，形成长期复用能力。

## 10. 当前初始化版本的边界

当前版本已完成的是：

- 工程骨架
- 基础 CLI
- 能力说明文档
- 工作流说明文档
- workspace 初始化能力

当前版本尚未直接内置的是：

- 重型音频模型依赖
- 实际 stem 分离模型
- 实际 pitch extraction 模型
- 乐谱导出引擎
- DAW 插件桥接

这些能力可以在你后续提出具体任务后，按实际需要逐步接入，而不是一次性过度设计。
